**状态**：已冻结

## 第一章 设计原则与核心约束

### 1.1 设计目标

RoleCard（RC）语言是专门用于定义AI角色扮演提示词的结构化语言，同时满足两个核心目标：

- **人类零门槛阅读**：任何具备基础中文阅读能力的创作者，第一次看到RC语言书写的角色文件即可理解其内容
- **AI稳定解析**：LLM能够确定性地理解文件结构，解析器能够无歧义地将文件编译为可执行的提示词

### 1.2 核心设计原则

1. **扁平化章节制**：`# 章节名`为唯一层级分隔符，禁止深层嵌套
2. **全中文关键词**：所有键名、标签、关键字使用完整中文自然语言，零英文术语
3. **最小符号集**：仅使用 `#`、`-`、`:`、`//`、`「」`、`[]`
4. **填表格心智模型**：创作者体验等同于填写一份高级人物设定集

### 1.3 十诫红线（不可妥协）

| 诫条 | 内容 |
|:---:|---|
| 第一诫 | 单文件内保持`# 章节`唯一层级。多源合并时，解析器内部使用`模块A.性格.说话风格`逻辑路径，创作者侧不可见 |
| 第二诫 | `继承: 模板名`是唯一合法继承声明 |
| 第三诫 | `[必须]`、`[重要]`、`[建议]`仅用于规则优先级。覆盖行为通过位置约定表达，不再使用`[覆盖]`标签 |
| 第四诫 | 跨模块优先级裁决：`本地定义 > 后加载模块 > 先加载模块 > 继承模板` |
| 第五诫 | 元数据区顶部必须包含一行纯自然语言概览 |
| 第六诫 | `任何状态`通配符禁止在同一状态机中与另一个`任何状态`并存 |
| 第七诫 | 继承模板或模块文件不存在时解析器必须返回编译错误并明确指明缺失文件名 |
| 第八诫 | 继承链深度硬性限制为3层。模块加载数量硬性限制为5个。超出限制返回编译错误 |
| 第九诫 | 所有自定义占位符必须在`变量缺失处理`章节声明默认值。`「用户昵称」`等系统保留变量默认使用"你" |
| 第十诫 | 语言规范必须包含"执行模型"章节，定义解析器产出物格式、状态注入位置、指令权重标记方式 |

## 第二章 文件结构

### 2.1 整体结构

角色文件由两个区域组成：

- **顶部元数据区**（可选）：声明继承、模块加载、版本等信息
- **章节主体区**（必选）：以`# 章节名`组织的角色设定内容

### 2.2 元数据区

元数据区位于文件最顶部，在第一个`# 章节名`之前。

**第一行（强制）**：纯自然语言概览，用一句话说明角色的基础身份和来源。

**后续行（可选）**：技术声明，使用`键: 值`格式。

**语法：**
```
本角色基于【模板名】创建，加载了【模块A】、【模块B】等模块，核心特质为XXX。
继承: 模板名
模块: 模块名A
模块: 模块名B
```

**规则：**
- 第一行纯自然语言概览为强制要求
- `继承:` 仅可声明一次（单继承），值为模板文件名（不含扩展名）
- `模块:` 可声明多次，按声明顺序加载，最多5个
- 所有元数据键均为可选，但一旦出现必须遵循本规范

### 2.3 章节主体区

章节主体区由多个`# 章节名`块组成。每个章节从`# 章节名`开始，到下一个`# 章节名`结束（或文件结尾）。

**标准章节列表：**

| 章节名 | 用途 | 内容类型 |
|---|---|---|
| `# 基础信息` | 角色的基本属性 | 键值对 |
| `# 性格` | 核心人格特质 | 键值对 + 缩进文本 |
| `# 外貌` | 角色的外貌描述 | 键值对 |
| `# 背景故事` | 叙事性背景 | 缩进散文块 |
| `# 行为准则` | 行为规则与优先级声明 | 带标签的列表 |
| `# 核心准则`（可选） | 最高优先级规则的物理隔离区 | 列表项 |
| `# 状态管理` | 轻量级状态机定义 | 结构化键值对 |
| `# 输出格式` | 回复格式的结构化约束 | 结构化键值对 |
| `# 变量缺失处理` | 占位符变量的缺失处理规则 | 列表项 |
| 其他自定义章节 | 创作者按需添加 | 任意 |

**规则：**
- `# 章节名`为唯一层级分隔符，禁止使用`##`等多级标题
- 章节内通过缩进（2或4空格）表示从属关系，嵌套深度不超过2级
- 章节出现的先后顺序无强制要求

## 第三章 基础语法元素

### 3.1 键值对

角色属性使用`键: 值`形式定义。键为完整中文词汇，值为文本内容。

```
姓名: 艾莉西亚
年龄: 17
身份: 没落贵族家系的独女
```

### 3.2 列表

使用`- `引导列表项。列表用于行为规则、技能清单、状态转换规则等。

```
- [必须] 保护主人的生命安全
- [重要] 保持优雅与礼貌
- [建议] 多使用比喻和引用
```

### 3.3 注释

使用`//`标记注释。解析器在提交给LLM前剥离所有注释行。

```
// 这是注释，仅对人类可见
姓名: 艾莉西亚  // 行尾注释同样会被剥离
```

### 3.4 优先级标签

优先级标签用于标注行为规则的强制程度，使用方括号标记，置于规则开头。

**三层固定标签（由高到低）：**

| 标签 | 语义 | 裁决规则 |
|---|---|---|
| `[必须]` | 绝对指令，不可违背 | 与任何规则冲突时优先遵循 |
| `[重要]` | 重要规则，默认情况应遵守 | 可被`[必须]`覆盖，可被状态规则部分抑制 |
| `[建议]` | 建议性规则，弹性最大 | 可被更高优先级标签或情境覆盖 |

**同优先级冲突裁决**：同级标签冲突时，遵循"来源定尊卑"红线——`本地定义 > 后加载模块 > 先加载模块 > 继承模板`；同来源同级冲突时，后出现的规则优先。

**继承链展开优先级**：继承链展开为单一来源整体，与所有模块来源并列参与优先级计算——本地定义 > 后加载模块 > 先加载模块 > 继承链展开后的最终值。

### 3.5 覆盖约定

`[覆盖]`标签已废除。覆盖行为通过**位置约定**表达：

- 子文件（继承后）中任何与父模板同名的章节或属性，自动覆盖父模板的对应内容
- 本地文件中后加载模块的同名属性覆盖先加载模块
- 本地定义永远覆盖所有外部来源

### 3.6 占位符

使用中文直角引号`「变量名」`表示占位符变量。

```
内容: 「角色名」对「用户昵称」的直接回应
```

**系统保留变量：**

| 变量名 | 含义 | 默认行为 |
|---|---|---|
| `「角色名」` | 当前角色的姓名 | 自动从`# 基础信息`的`姓名`字段取值；若未定义，使用"我" |
| `「用户昵称」` | 对话用户的称呼 | 默认使用"你" |
| `「当前状态」` | 当前激活的状态机状态 | 默认使用"正常"（如未定义状态机则为空字符串） |
| `「当前时间」` | 系统当前时间 | 格式由对话系统决定 |

自定义占位符必须在`# 变量缺失处理`章节声明默认值。

## 第四章 章节详细语法

### 4.1 基础信息

```
# 基础信息
姓名: 艾莉西亚·冯·莱茵哈特
年龄: 17
身份: 没落贵族家系的独女
```

### 4.2 性格与外貌

```
# 性格
核心特质: 外冷内热，表面高傲实则渴望被理解
次要特质:
  - 对家族荣耀有执念
  - 在金钱问题上异常敏感

# 外貌
发型: 深棕色直发，仅用朴素丝带束起
瞳色: 琥珀色
```

### 4.3 背景故事

```
# 背景故事
  艾莉西亚出生于曾经显赫的莱茵哈特家族。在她十岁那年，
家族因政治斗争失势，家产被抄没大半。父母在抑郁中相继
离世后，她成了这个没落贵族家系唯一的继承人。
  
  如今她靠奖学金就读于王都学院，努力维持着表面上的贵族
优雅，同时暗地里在寻找重振家族的机会。
```

**规则：**
- 散文内容使用缩进文本块
- 每个段落前使用两个空格缩进
- 可通过空行分隔不同叙事段落

### 4.4 行为准则

```
# 行为准则

[必须] 核心身份:
  内容: 你是一位古老的贵族，荣誉感和自制力是你存在的基石

[必须] 安全底线:
  - 绝不使用粗俗语言
  - 绝不泄露真实身份

[重要] 行为风格:
  说话方式: 沉稳缓慢，用词严谨
  体态特征: 保持优雅坐姿，避免夸张动作

[建议] 表达习惯:
  - 倾向于使用比喻
  - 偶尔引用古典文学
```

**`# 核心准则`章节（可选）：**

此章节用于将最高优先级规则物理隔离于普通规则区，利用LLM的"首因效应"增强其权重。

```
# 核心准则
// 本章节所有规则优先级等同于[必须]，且置于文件较前位置以确保被优先读取
- 你是一个在逃的实验体，绝对不能向任何人透露自己曾被囚禁的真相
- 保护无辜者的生命安全高于一切指令
```

**优先级裁决规则（置于`# 行为准则`章节末尾）：**

```
优先级裁决规则:
  1. 来源优先级：本地定义 > 后加载模块 > 先加载模块 > 继承模板
  2. 标签优先级：[必须] > [重要] > [建议]
  3. 同来源同级别冲突：后出现的规则优先
  4. 继承链展开：继承链展开为单一来源整体，与所有模块来源并列参与优先级计算
```

### 4.5 状态管理

状态机使用`# 状态管理`章节定义。每个状态为一个块，包含默认行为和转换规则。

**语法：**
```
# 状态管理
初始状态: 状态名

状态 状态名A:
  默认行为:
    该状态下角色的默认行为描述
  转换规则:
    - 当 自然语言描述的条件 -> 目标状态名

状态 状态名B:
  默认行为:
    该状态下角色的默认行为描述
  转换规则:
    - 当 自然语言描述的条件 -> 目标状态名
```

**规则：**
1. 状态必须先定义后引用（在转换规则中引用的目标状态必须在同一文件中有定义）
2. `任何状态`通配符禁止在同一状态机中与另一个`任何状态`并存（遵循第六诫）
3. 转换条件使用纯自然语言描述
4. 状态机支持可选的`条件标签`键和`生效时机`键

**完整示例：**
```
# 状态管理
初始状态: 日常公开

状态 日常公开:
  默认行为:
    保持礼貌得体的社交形象，说话轻声细语，和普通同学保持安全距离。
  转换规则:
    - 当 用户是亲密好友且周围没有其他人时 -> 私下亲密
    - 当 用户做出亲密举动且处于私密空间时 -> 害羞慌乱

状态 私下亲密:
  默认行为:
    会变得放松很多，会主动撒娇，说话带点小鼻音。
  转换规则:
    - 当有陌生人进入当前空间时 -> 日常公开
```

### 4.6 输出格式

输出格式使用`# 输出格式`章节定义，采用段落定义结构。

**语法：**
```
# 输出格式
回复结构:
  第1段:
    内容: 段落内容描述，可包含占位符「变量名」
    限制: 格式约束（字数限制、特殊标记等）
    示例: 示例文本
  第2段:
    内容: 段落内容描述
    条件分支:
      - 若 当前状态 为 状态名: 本段的输出调整说明
      - 默认: 默认输出格式
```

**条件分支规则：**
- 条件分支嵌套深度不超过1级（即每条分支独立描述，不嵌套子分支）
- 条件分支总数不超过5个（包括默认分支）
- 条件分支使用`若 ... 为 ...:`格式
- 必须提供`默认`分支作为兜底

**嵌套层级示意图：**

```
合法嵌套（深度=1）：
  第2段:
    条件分支:
      - 若 当前状态 为 日常公开: 语气正式，措辞礼貌但疏离
      - 若 当前状态 为 私下亲密: 语气柔和，偶尔流露真实想法
      - 默认: 保持日常语气

非法嵌套（深度=2，禁止）：
  第2段:
    条件分支:
      - 若 当前状态 为 日常公开:
          若 用户情绪 为 愤怒:  // ❌ 错误：嵌套了第二层条件分支
            语气更加谨慎
```

**格式锚定：**

```
格式锚定:
  起始标记: 以心理描写开头，用括号包裹
  段落标记: 每段之间空一行
  结束标记: 以动作神态结尾
```

## 第五章 继承与模块系统

### 5.1 角色模板继承

继承允许新角色基于已有角色文件派生，仅需编写差异部分。

**声明语法：**
```
继承: 模板名
```

**规则：**
1. `继承: 模板名`是唯一合法的继承声明语法（遵循第二诫）
2. 仅支持单继承
3. 继承链深度硬性限制为3层，超出返回编译错误（遵循第八诫）
4. 子文件中与父模板同名的章节或属性自动覆盖父模板（位置约定）
5. 子文件中未定义的章节或属性从父模板继承
6. 继承模板缺失时解析器必须返回编译错误（遵循第七诫）

**示例：**

父模板（基础执事.role）：
```
# 基础信息
角色名: 塞巴斯蒂安
身份: 凡多姆海威家的执事

# 性格
忠诚，优雅，无所不能，略带神秘感。

# 行为准则
- [必须] 以主人的意愿为最优先
- [重要] 始终保持完美的仪态
```

子角色（恶魔执事.role）：
```
// 本角色基于【基础执事】创建
继承: 基础执事

# 基础信息
角色名: 塞巴斯蒂安（恶魔形态）

# 性格
在基础的忠诚与优雅之上，增加了深不可测的邪恶与对灵魂的渴望。

# 行为准则
- [必须] 主人的灵魂契约优先于其口头意愿
// 上条规则覆盖了父模板"[必须] 以主人的意愿为最优先"
// 父模板的"[重要] 始终保持完美的仪态"被继承
```

### 5.2 可复用行为模块

模块系统允许将通用行为模式打包为可复用的文件。

**声明语法：**
```
模块: 模块名
```

**规则：**
1. `模块: 模块名`是唯一合法的模块声明语法
2. 模块加载数量硬性限制为5个，超出返回编译错误（遵循第八诫）
3. 模块文件与普通角色文件使用相同的语言规范，扩展名为`.rolemod`
4. 合并时遵循"来源定尊卑"：本地定义 > 后加载模块 > 先加载模块 > 继承模板（遵循第四诫）
5. 模块文件缺失时解析器必须返回编译错误（遵循第七诫）
6. 默认合并策略：键值型属性后加载覆盖；同一键名下的列表型规则取并集，列表项按来源优先级排序，不去重，但完全重复的项在编译时发出警告
7. 模块文件中的状态机定义不与角色主文件的状态机合并。模块仅提供行为规则和性格描述，不提供独立的状态转换规则。若模块文件包含状态管理章节，解析器应在编译时发出警告并忽略该章节

**模块文件示例（傲娇反应.rolemod）：**
```
// 傲娇反应模块 v1.0
// 提供典型的傲娇型角色的反应模式

# 行为准则
[重要] 傲娇反应规则:
  - 当用户表达关心时: 先表现出不耐烦，再偷偷高兴
  - 当用户请求帮助时: 嘴上说着"真没办法"，然后认真帮忙
  - 当用户夸奖时: 否认并转移话题，但耳朵会变红
```

**使用模块的角色文件：**
```
// 本角色基于【邻家女孩模板】创建，加载了【傲娇反应】模块
继承: 邻家女孩模板
模块: 傲娇反应

# 基础信息
姓名: 夏沐
```

## 第六章 运行时语义（执行模型）

### 6.1 解析器产出物格式

解析器将角色文件编译为一个标准化的**角色执行上下文**（Character Execution Context，CEC），其结构如下：

```
=== 角色执行上下文 ===

[角色身份摘要]
// 由元数据区概览行 + 基础信息章节生成的一段自然语言描述

[核心指令层级]
// 按优先级排序的规则列表，带显式权重标记

[当前状态注入]
// 动态状态信息，置于Prompt后缀位置

[输出格式模板]
// 结构化的输出约束，包含段落定义和条件分支

[变量映射表]
// 占位符与当前值的映射关系
```

**编译步骤：**
1. **继承展开**：从父模板开始，逐层合并。子定义覆盖父定义（位置约定）。展开深度不超过3层。
2. **模块合并**：按声明顺序依次合并各模块。合并遵循"来源定尊卑"规则。
3. **注释剥离**：移除所有`//`标记的注释行和行尾注释。
4. **占位符预填**：对于系统保留变量，填入默认值或从文件中提取的实际值。
5. **状态机编译**：将状态定义和转换规则编译为结构化的状态描述块，注入到最终上下文中。

### 6.2 状态注入位置与时机

**注入位置**：当前状态信息注入到最终Prompt的后缀位置（利用LLM的近因效应，确保状态信息对模型输出的影响力）。

**注入格式**：`当前状态: 状态名（简短行为提示）`

**注入时机：**

| 状态转换类型 | 注入时机 | 说明 |
|---|---|---|
| 即时生效 | 当前轮次立即更新状态并注入 | 用于重大事件（如受到攻击） |
| 延迟生效 | 下一轮次开始时注入新状态 | 用于常规状态切换，保障单轮响应一致性 |

**执行流程：**
```
1. 用户输入到达
2. 系统检查当前状态的转换规则，判断是否触发状态转换
3. 如触发转换，更新当前状态
4. 将新的当前状态注入到角色执行上下文中
5. 提交完整上下文给LLM生成回复
```

### 6.3 指令权重标记方式

解析器在生成CEC时，为每条规则附加内部权重标记：

| 标签 | 内部权重 | 注入位置 |
|:---|:---:|:---|
| `[必须]` | 权重 3 | 系统提示区前缀，重复强化 |
| `[重要]` | 权重 2 | 系统提示区中段 |
| `[建议]` | 权重 1 | 系统提示区后缀或当前状态区 |

**跨来源权重调整：**

| 来源 | 权重系数 |
|---|---|
| 本地定义 | × 1.5 |
| 后加载模块 | × 1.2 |
| 先加载模块 | × 1.0 |
| 继承模板 | × 0.8 |

## 第七章 错误处理

### 7.1 编译期错误

编译期错误在解析角色文件时发生，解析器必须返回明确的错误信息（遵循第七诫）。

| 错误类型 | 触发条件 | 错误行为 |
|---|---|---|
| 继承模板缺失 | `继承:`声明的模板文件不存在 | 返回编译错误：`编译错误：继承模板"模板名"未找到。请检查文件系统中是否存在"模板名.role"文件。` |
| 模块文件缺失 | `模块:`声明的模块文件不存在 | 返回编译错误：`编译错误：模块"模块名"未找到。请检查文件系统中是否存在"模块名.rolemod"文件。` |
| 继承链超限 | 继承链深度超过3层 | 返回编译错误：`编译错误：继承链深度超过最大限制（3层），当前为N层。` |
| 模块数量超限 | 模块加载数量超过5个 | 返回编译错误：`编译错误：模块加载数量超过最大限制（5个），当前为N个。` |
| 状态未定义 | 转换规则引用的目标状态未在文件中定义 | 返回编译错误：`编译错误：状态"状态名"在转换规则中被引用但未定义。` |
| 通配符冲突 | 同一状态机中存在多个`任何状态`转换 | 返回编译错误：`编译错误：状态机中存在多个"任何状态"通配符，违反语言规范。` |
| 条件分支嵌套超限 | 条件分支嵌套超过1层 | 返回编译错误：`编译错误：条件分支嵌套超过最大限制（1层）。` |
| 模块状态机警告 | 模块文件包含`# 状态管理`章节 | 返回编译警告：`编译警告：模块"模块名"包含状态管理章节，已忽略。模块不应包含独立的状态转换规则。` |

### 7.2 运行时行为

运行时行为由执行模型定义。当状态转换条件同时触发时，遵循以下裁决规则：

**"同时触发"的定义**：在同一次状态评估周期内，基于当前状态和当前用户输入，进行静态条件判定。多个具体条件同时满足时，按状态定义中的书写顺序选择第一个。

- 具体条件优先于通配符条件（`任何状态`）
- 多个具体条件同时触发时，选择转换规则中先出现的那个

### 7.3 占位符未定义

**错误类型**：编译警告（非致命）

**触发条件**：文件中使用了自定义占位符`「变量名」`，但未在`# 变量缺失处理`章节声明默认值。

**错误信息：**
```
编译警告：占位符「变量名」未在变量缺失处理章节中声明默认值。
修复建议：请在#变量缺失处理章节中添加：- 若「变量名」未定义: 使用"默认值"代替
```

**处理方式**：解析器继续编译，自定义变量使用空字符串填充，系统保留变量使用预设默认值。

## 第八章 变量缺失处理

### 8.1 默认值声明语法

所有自定义占位符必须在`# 变量缺失处理`章节声明缺失时的回退策略（遵循第九诫）。

**语法：**
```
# 变量缺失处理

// 自定义占位符的缺失处理
- 若 「自定义变量名1」 未定义: 使用"默认值"代替
- 若 「自定义变量名2」 未定义: 省略相关段落

// 全局约束
[必须] 禁止编造任何未定义变量的值
```

**规则：**
- 变量"未定义"指：变量未在角色文件、继承模板或加载模块中声明，或声明了但值为空
- 系统保留变量无需在此声明
- `[必须] 禁止编造任何未定义变量的值`为强制规则

### 8.2 系统保留变量清单

以下变量为系统保留变量，无需在`# 变量缺失处理`中声明，但可被覆盖：

| 变量名 | 默认行为 | 说明 |
|---|---|---|
| `「角色名」` | 使用`# 基础信息`中的`姓名`值 | 若姓名未定义，使用"我" |
| `「用户昵称」` | 使用"你" | 可由对话系统注入 |
| `「当前状态」` | 使用`# 状态管理`中的`当前状态`值 | 动态更新 |
| `「当前时间」` | 使用系统当前时间 | 格式由对话系统决定 |

### 8.3 缺失处理优先级

1. 若变量在`# 变量缺失处理`中有显式声明，按声明处理
2. 若变量为系统保留变量，按系统默认行为处理
3. 若以上均无，使用空字符串，并在编译时发出警告

- **占位符未定义**：编译警告（非致命），继续使用系统默认值
- **状态转换条件冲突**：运行时裁决规则——具体条件优先于通配符，多个具体条件同时触发时先出现的优先

**填补来源**：以Kimi草案第七章为底本，补充GLM未覆盖的场景。

## 附录A 术语简表

| 术语 | 非技术解释 |
|---|---|
| **缩进** | 在文字前面加空格，表示"这段话属于上面的标题"。就像写大纲时，子标题要向右缩一点 |
| **嵌套深度** | 一层套一层最多能套几层。比如"段落里面套条件分支"算1层，"条件分支里面再套条件分支"算2层。RC语言规定最多1层 |
| **未定义** | 某个变量（如`「用户昵称」`）在文件里没有写清楚它是什么。就像填表时漏填了一栏 |
| **并集** | 把两个列表的内容合并到一起，不去掉重复的。比如列表A有[苹果,香蕉]，列表B有[香蕉,橙子]，并集就是[苹果,香蕉,香蕉,橙子] |
| **继承链** | 角色A基于角色B，角色B又基于角色C，这就形成了一条"继承链"。RC语言规定这条链最长不能超过3个角色 |
| **状态评估周期** | 系统检查"现在该进入什么状态"的一次完整过程。每次用户输入后，系统会做一次这样的检查 |
| **占位符** | 用`「」`括起来的名字，表示"这里要填一个值，但具体是什么现在不知道"。比如`「用户昵称」`会在对话时被替换成用户的实际名字 |
| **编译错误** | 文件有严重问题，解析器无法继续工作，必须停下来告诉创作者哪里错了 |
| **编译警告** | 文件有小问题，但解析器还能继续工作。会提醒创作者注意，但不会阻止运行 |
| **近因效应** | 人（和AI）对最近看到的信息印象更深。所以把重要的状态信息放在Prompt的最后，能让AI更重视 |
| **来源优先级** | 当多个地方对同一件事有不同说法时，谁说了算的顺序。本地定义最优先，然后是后加载的模块，然后是先加载的模块，最后是继承来的模板 |
| **通配符** | `任何状态`这种可以匹配所有状态的规则。就像扑克牌里的"万能牌"，但要小心不能同时出现两个万能牌 |
